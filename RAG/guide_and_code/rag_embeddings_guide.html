🤖 RAG & Embeddings

Complete Beginner's Guide to Retrieval-Augmented Generation

🎯What is RAG?

RAG (Retrieval-Augmented Generation) is a revolutionary approach that combines the power of large language models with external knowledge retrieval systems.

💡 Simple Analogy: Think of RAG like an open-book exam. Instead of relying solely on what the AI "remembers" from training, it can look up relevant information from external documents and then provide a more accurate, contextual answer.

How RAG Solves Real Problems

✅ Problems RAG Solves

  • Knowledge cutoff limitations
  • Hallucinations and inaccurate information
  • Lack of domain-specific knowledge
  • Need for real-time information

⚠️ RAG Challenges

  • Increased complexity and latency
  • Quality depends on retrieval accuracy
  • Requires good document preprocessing
  • Additional infrastructure needed

✂️Document Chunking Strategy

Since LLMs have context window limitations (typically 4K-32K tokens), we need to break large documents into manageable pieces.

📝 Chunking Example

Original Document:

"Refunds are processed within 5 business days. Customers must apply using the portal. Our support team reviews each request carefully."

After Chunking (10 words each):

Chunk 1: "Refunds are processed within 5 business days." Chunk 2: "Customers must apply using the portal." Chunk 3: "Our support team reviews each request carefully."

Overlap: The Secret Sauce

Overlap prevents context from being lost at chunk boundaries by repeating a few words between adjacent chunks.

🔄 Overlap Example (Chunk size=5, Overlap=2)

Text: "Refunds are processed within 5 business days via the portal."

Chunk 1: "Refunds are processed within 5" Chunk 2: "within 5 business days via" (overlaps: "within 5") Chunk 3: "days via the portal" (overlaps: "days via")
💡 Pro Tip: A good overlap ratio is 10-20% of your chunk size. This ensures context continuity without excessive redundancy.

  2. Similarity Search: Find most relevant document chunks using cosine similarity
  3. Context Assembly: Combine retrieved chunks with the original question
  4. Generation: LLM generates answer using both its knowledge and retrieved context

🔍 Why We Need Embedding Models

Traditional keyword search looks for exact text matches. Embedding models understand semantic meaning:

  • "How to reset password?" ≈ "Password recovery steps"
  • "Dog" ≈ "Canine" ≈ "Puppy"
  • "Happy" ≈ "Joyful" ≈ "Delighted"

📊The K Parameter in RAG

K represents the number of most relevant chunks to retrieve from your vector database.

🔍 K=3 Example

Question: "How to reset my password?"

Vector DB contains 1000 chunks, but only top 3 most similar are retrieved:

Chunk #247: "To reset your password, go to settings..." (Similarity: 0.89) Chunk #891: "Password recovery requires email verification..." (Similarity: 0.85) Chunk #156: "Forgot password? Click the reset link..." (Similarity: 0.82)

Choosing the Right K Value

  • K too small (1-2): Might miss important context
  • K too large (10+): Adds noise, exceeds context limits
  • Sweet spot (3-7): Balances relevance and context size

🎯Optimal Chunk Size: The 500-Token Sweet Spot

The Goldilocks Principle

Just like in the fairy tale, chunk size needs to be "just right" - not too small, not too big.

✅ Chunk Size ~500 Tokens

  • Contains complete thoughts/paragraphs
  • Maintains context integrity
  • Fits well in LLM context windows
  • Good balance of specificity and scope

❌ Too Small (<200 tokens)

  • Fragments important information
  • Loses context and meaning
  • Requires more chunks for same info

❌ Too Large (>1000 tokens)

  • Contains irrelevant information
  • Wastes LLM context space
  • Reduces retrieval precision
💡 Industry Standards:
  • Technical docs: 400-600 tokens
  • Legal documents: 600-800 tokens
  • General content: 300-500 tokens

📏Text Processing Mathematics

Understanding Token Calculation

Average English Word ≈ 5 characters
With spaces/punctuation ≈ 6-7 characters per word
Tokenization ratio ≈ 1.3 tokens per word
Therefore: 500 tokens ≈ 385 words ≈ 2,300 characters

📊 Real-World Examples

  • Short paragraph: ~50-100 tokens
  • Medium paragraph: ~100-250 tokens
  • Long paragraph: ~250-400 tokens
  • Page of text: ~500-800 tokens

🔢Model Parameters Demystified

What Does "7B" or "70B" Actually Mean?

These numbers represent the total trainable parameters (weights) in the neural network.

Total Parameters ≈ Embedding Matrix + Transformer Layers + Output Layer

Example for 7B Model:
Vocabulary (50K) × Embedding Dim (4096) = 204M parameters
+ Transformer Layers ≈ 6.8B parameters
= ~7 Billion total parameters

💾 Storage & Memory Impact

  • 7B model: ~14GB RAM (FP16) or ~28GB (FP32)
  • 13B model: ~26GB RAM (FP16) or ~52GB (FP32)
  • 70B model: ~140GB RAM (FP16) - requires multiple GPUs

⚡FAISS: Vector Search Engine

FAISS (Facebook AI Similarity Search) is Meta's open-source library for efficient similarity search and clustering of dense vectors.

Why FAISS is Popular

✅ FAISS Advantages

  • Extremely fast similarity search
  • Scales to millions of vectors
  • Multiple index types for different use cases
  • CPU and GPU support
  • Memory efficient

⚠️ FAISS Considerations

  • Requires technical setup
  • Limited built-in persistence
  • No distributed features out of the box
# Basic FAISS Usage import faiss import numpy as np # Create index for 1536-dimensional vectors dimension = 1536 index = faiss.IndexFlatL2(dimension) # Add vectors to index vectors = np.random.random((1000, dimension)).astype('float32') index.add(vectors) # Search for similar vectors query = np.random.random((1, dimension)).astype('float32') distances, indices = index.search(query, k=5)

FAISS Index Types

  • IndexFlatL2: Exact search, good for small datasets (<1M vectors)
  • IndexIVFFlat: Approximate search, faster for large datasets
  • IndexHNSW: Graph-based, excellent for high-dimensional data
  • IndexPQ: Product quantization, very memory efficient

🚀Advanced RAG Techniques

Hybrid Search

Combines semantic search (embeddings) with keyword search (BM25) for better results.

Final Score = α × Semantic Score + (1-α) × Keyword Score
where α is typically 0.7-0.8

Re-ranking

Use a cross-encoder model to re-rank the top-K results for better precision.

Multi-hop Reasoning

Perform multiple retrieval steps to answer complex questions requiring information from different sources.
